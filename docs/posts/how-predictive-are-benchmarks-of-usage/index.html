How Can We Predict Which Models People Use?

June 12, 2026 1892 words 9 min read

The point of a model benchmark is to help answer a practical question: which model should I use?

That is easy to forget because benchmarks often get discussed as leaderboards. A benchmark can look rigorous, have hard tasks, and still fail at the thing users need from it: helping them pick the model that will work best for their use-cases.

One way to test whether a benchmark is useful is to ask whether it predicts revealed preference. When two models are available at roughly the same price, does the model with the higher benchmark score get used more? If the answer is usually yes, the benchmark is probably measuring something users value. If the answer is no, either the benchmark is missing the real capability, or other factors such as latency or distribution are dominating.

This matters from both sides of the market. Users need to know which benchmarks actually help them pick a model for a use case. Labs need to know which capabilities users reward, and how pricing changes might trade off margin against market share. If a model is highly capable but overpriced relative to substitutes, usage should reveal that. If a model is cheap but underused, that says something different.

This framing is useful because OpenRouter exposes unusually good public market data: model prices, token usage, and request counts. I already have a benchmark dashboard with normalized model IDs and benchmark scores, so the implementation is mostly a join between those two datasets.

Method

I pull:

  • OpenRouter’s public model catalog from the Models API, including prompt and completion pricing.
  • OpenRouter’s public rankings endpoints, including current day/week/month/trending token usage.
  • My existing benchmark data, including the aggregate capabilities index, category sub-indices, and individual benchmark scores.

For the main score, I use paid OpenRouter usage over the month view. Free variants are excluded from the primary analysis.

The price variable is an effective observed price, not just the listed input-token price or output-token price. Models have very different input/output mixes, so I estimate dollars per million served tokens from the observed token mix in OpenRouter’s ranking rows:

estimated_cost =
  prompt_tokens * prompt_price
  + completion_tokens * completion_price
  + cached_tokens * cache_read_price

served_tokens =
  prompt_tokens + completion_tokens + reasoning_tokens

effective_price_per_million =
  estimated_cost / served_tokens * 1,000,000

If OpenRouter does not expose a cache-read price, I fall back to the prompt-token price for cached tokens. This effective price is still approximate, but it is much closer to the user’s actual economic tradeoff than comparing models by input price alone.

The analysis then runs in three passes.

First, I estimate the price baseline:

log_usage = a + b * log_effective_price

This answers the simplest question: how much of model usage is explained by price before looking at benchmark scores at all?

Second, I estimate raw score-to-usage relationships for each benchmark:

log_usage = a + b * benchmark_score

This asks whether a benchmark’s scores line up with usage in the raw market data. I report both Pearson correlation against log usage and Spearman rank correlation.

Third, I ask whether scores explain usage after accounting for price:

log_usage = a + b * log_effective_price + c * benchmark_score

The key statistic here is incremental R^2: how much more usage variance is explained when benchmark score is added to the price-only model. I also report a partial correlation, computed by residualizing both usage and score against price and then correlating the residuals:

partial_r_after_price =
  corr(residual(log_usage ~ log_price), residual(score ~ log_price))

Finally, for each benchmark, I also look at every pair of matched models whose effective prices are within 2x of each other. A benchmark gets a correct prediction when the model with the higher benchmark score also has higher OpenRouter token usage. This gives a simple price-controlled pairwise accuracy:

pairwise_accuracy = correct same-price-ish model orderings / all same-price-ish model orderings

So the analysis starts with the obvious confounder, price; then asks whether scores correlate with usage at all; then asks whether scores add explanatory power beyond price.

This does not claim usage is capability. Usage is a messy outcome variable. But that is the point: if a benchmark cannot predict real model choice even after roughly controlling for price, I should be suspicious about using it as a proxy for practical value.

First Read

Price matters, but it is not the whole story. Across the current paid OpenRouter sample, log(effective price) and log(usage) have a correlation of about -0.30. The log-log price elasticity is about -0.60, meaning a 10% higher effective price is associated with roughly 6% lower usage in this cross-sectional cut. Price alone explains about 9% of log usage variance.

The price bands are useful context. Models below $0.50 per million effective tokens account for about two thirds of paid token volume, but the $2-$10 band still accounts for about one fifth because high-end models like Claude Sonnet and Opus get substantial use despite much higher prices. The very expensive $10+ band has little token share.

Raw benchmark correlations are stronger than the price baseline. The agentic capabilities index has a raw score-to-log-usage correlation of about 0.57; the all-up capabilities index is about 0.45. Among individual benchmarks, HiL-Bench, GSO-Bench, Terminal-Bench Hard, ProgramBench Extended, and Humanity’s Last Exam are near the top of the raw score-to-usage table in this scrape.

The more important result is the price-adjusted one. Adding the all-up capabilities index to a price-only usage model adds about 50 percentage points of R^2 on its matched model set. The agentic index adds about 54 percentage points. GSO-Bench and Terminal-Bench Hard also add large amounts of explanatory power after price. This is the signal I care about most: benchmark score is not just rediscovering that cheap models get used.

Pairwise accuracy is a different lens on the same question. In the current scrape, my all-up capabilities index gets about 75% of price-matched model pairs right. The agentic sub-index does better, at about 79%, which fits the current OpenRouter user base: a lot of usage appears to come from coding, automation, and agent-style workflows rather than generic chat.

Several individual benchmarks look surprisingly predictive by pairwise accuracy. EQ-Bench, GSO-Bench, Terminal-Bench Hard, CritPt, GDPval-AA, PPBench, and ProofBench all clear 74% pairwise accuracy in this scrape. I would not overinterpret the exact ordering because coverage differs a lot by benchmark, but the broad pattern is useful: benchmarks with open-ended or work-like tasks tend to predict usage better than narrow one-off tasks.

The weak end is also informative. SlopCodeBench, FrontierCode, Blueprint-Bench 2, HalluHard, DeepSWE, and Kaggle Game Arena all come in near chance or below in this cut. That does not mean they are bad benchmarks in isolation. It means they are not currently explaining OpenRouter paid usage at a given price very well. Some of that is probably coverage and timing. Some of it may be that they measure a narrower slice than the user base is buying.

Pricing Implications

For users, the benchmark side and the price side are inseparable. A model with a higher score is not automatically the best choice; it has to justify its price for the task. That is why the analysis asks whether benchmark scores still explain usage after price. A benchmark that survives that test is more likely to be useful when choosing between similarly priced models.

For labs, the same data can be turned around. If usage is strongly price-sensitive and a model is losing share to similarly capable cheaper substitutes, the model may be priced above the market-clearing range for that audience. If a model has strong benchmark-adjusted demand despite a high price, there may be room to preserve margin.

The interactive pricing calculator uses the price-and-capability model directly. Select a model, then adjust its input price, output price, input/output token ratio, effective price, market share, or capabilities index.

Input price, output price, and token mix define the effective price:

effective_price =
  (input_price * input_output_ratio + output_price)
  / (input_output_ratio + 1)

Changing the input price, output price, or input/output ratio updates effective price. Changing effective price scales input and output prices together while preserving their current ratio.

Market share and capability are linked through the capability-adjusted demand model:

log(market_share) =
  anchored_intercept
  + price_coefficient * log(effective_price)
  + capability_coefficient * capability_index

The important detail is the intercept. For each selected model, I recompute the intercept so the curve passes exactly through that model’s observed OpenRouter share, effective price, and capabilities index. That makes the calculator local to the selected model instead of pretending there is one universal intercept for every product and distribution channel.

Changing market share therefore solves for the effective price implied by the anchored curve. Changing capability keeps the current effective price fixed and recomputes the expected market share. The chart shows the estimated market-share curve over effective price at the current capability level.

What Makes A Benchmark “Good” Here?

This framing rewards benchmarks that preserve the same ordering users discover through actual use. A good benchmark should satisfy a few conditions:

  • It should distinguish models that users see as substitutes. A benchmark that only separates toy models from frontier models is not very useful once all serious models are above the bar.
  • It should transfer to a buying decision. If a model wins the benchmark but users prefer a similarly priced alternative, the benchmark may be missing instruction following, latency, reliability, context behavior, tool use, or output style.
  • It should have enough model coverage to make pairwise comparisons meaningful. A high score on 12 overlapping models is suggestive; a high score on 100 overlapping models is much stronger.
  • It should survive price controls. Raw usage mostly tells you that cheap models get used. The more interesting signal is whether benchmark score still predicts usage among similarly priced models.

The last point is why I prefer pairwise accuracy to a simple raw correlation. A single viral free model or a newly launched flagship can dominate raw token volume and distort correlation. Pairwise comparisons within a price band are less elegant statistically, but they match the real decision: “given my budget, which model should I call?”

Caveats

OpenRouter usage is not the whole market.

Usage also includes supply-side effects. Free previews, model defaults, provider outages, latency, context length, and distribution can move token share without saying much about model quality. Price is only one control, and my effective price estimate does not yet include every billing nuance.

Still, I like this test because it is adversarial to benchmarks in the right way. It asks whether the scoreboard predicts what people actually do when money is involved. A benchmark that fails that test is not automatically useless, but it owes us a more specific story about what it measures and when that measurement matters.
